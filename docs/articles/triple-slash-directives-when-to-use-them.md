# Triple-Slash Directives: When to Use Them (and When Not To)

**TL;DR**: `/// <reference types="..." />` is a per-file escape hatch for type inclusion. **The tsconfig `types` array is almost always the right tool instead.** Use triple-slash only when you can't touch the tsconfig, or when you're writing `.d.ts` declaration files.

---

## The Problem That Started This

We had a Cloudflare Workers package. To make wrangler's generated types available — things like `KVNamespace`, `DurableObjectNamespace`, our `Env` interface — wrangler's `typegen` command produces a `worker-configuration.d.ts` file and you wire it up in tsconfig like this:

```json
{
  "compilerOptions": {
    "types": ["./worker-configuration.d.ts"]
  }
}
```

That works great. Worker types load. Everyone's happy.

Then we added `env.ts` — a CLI utility that reads `wrangler.toml` and `.dev.vars` to give drizzle-kit and better-auth the database URL they need. It runs under Bun, not wrangler. So it uses `Bun.TOML.parse()` and `Bun.file()`:

```typescript
// apps/api/src/env.ts
const WranglerToml = type('string')
  .pipe((s) => Bun.TOML.parse(s))  // ← needs @types/bun
  .to({
    hyperdrive: [HyperdriveEntry, '...', HyperdriveEntry.array()],
  });

const tomlString = await Bun.file(
  new URL('../wrangler.toml', import.meta.url),
).text();  // ← needs @types/bun
```

TypeScript immediately complained. `Bun` is not defined. We had `@types/bun` installed as a dependency, but it wasn't being included. Why?

---

## The Gotcha: `types` Is an Exclusive Allowlist

Here's the thing people get tripped up on. The `types` field in tsconfig does not *add* to the default set of included types. It *replaces* it.

When you omit `types` entirely, TypeScript auto-includes every `@types/*` package it finds in `node_modules`. That's the default, open behavior.

The moment you write an explicit `types` array, TypeScript switches to strict mode: **only what you list is included, nothing else**. The array becomes an allowlist.

So this config:

```json
{
  "compilerOptions": {
    "types": ["./worker-configuration.d.ts"]
  }
}
```

...is silently saying "include the worker config types, and exclude everything else." Including `@types/bun`, even though it's sitting right there in `node_modules`.

Installing a `@types/*` package is not enough. If your tsconfig has an explicit `types` array, you have to list the package there too.

---

## The Wrong Fix: Sprinkling Triple-Slash Directives

The first instinct when TypeScript can't find your types is to add a triple-slash directive to the top of the file:

### The Workaround

```typescript
/// <reference types="@types/bun" />

import { type } from 'arktype';
// ... rest of file
```

This works. Triple-slash directives are per-file type inclusion instructions. They predate `tsconfig.json`'s `types` field and were the original way to pull in type declarations. TypeScript reads the comment, loads the declarations, and the `Bun` global is now available in that file.

But it's the wrong tool here. You've now created an implicit dependency that lives in a source file instead of the project configuration. The next person who reads `env.ts` has to know that comment isn't decorative — it's load-bearing. And if another file in the project ever needs Bun types, you'll be copy-pasting that directive instead of fixing the root cause.

---

## The Right Fix: Update the `types` Array

The correct answer is to add `@types/bun` to the tsconfig `types` array:

### The Canonical Fix

```json
{
  "compilerOptions": {
    "types": ["@types/bun", "./worker-configuration.d.ts"]
  }
}
```

That's it. Now `Bun` is available everywhere in the project, the Worker types are available everywhere in the project, and there are no magic comments in your source files.

You might wonder: does adding `@types/bun` conflict with the Workers globals? In practice, no. The Workers types come from `worker-configuration.d.ts` (generated by `wrangler types`). Bun's types come from `@types/bun`. They live in different namespaces. You get both.

---

## So When Does Triple-Slash Actually Make Sense?

There are three legitimate cases.

**1. Declaration files referencing other declaration files.**

When you're writing a `.d.ts` file and it needs to reference types from another declaration file, triple-slash is the right mechanism:

```typescript
// my-library.d.ts
/// <reference types="node" />

export declare function readConfig(path: string): NodeJS.ProcessEnv;
```

This is what the format was designed for. You're authoring types, not consuming them from a project, so there's no tsconfig in scope to configure.

**2. Shared or inherited tsconfig you can't modify.**

If you're working in a monorepo where the tsconfig is owned by another team, or you're extending a base config that's published as a package, and you can't add to the `types` array — a triple-slash in the specific file that needs it is reasonable. It's not pretty, but it's contained.

**3. Truly one-off inclusion in a large project with conflicts.**

If including a type package globally would genuinely break other files — because it introduces conflicting global declarations — and only one file actually needs it, a per-file directive makes the tradeoff explicit. This is rare. Most `@types/*` packages are designed not to conflict.

---

## The Decision Table

| Situation | Use |
|---|---|
| Whole project needs the types | `tsconfig` `types` array |
| Most files need the types | `tsconfig` `types` array |
| Can't modify tsconfig | Triple-slash directive |
| Writing a `.d.ts` file | Triple-slash directive |
| One file, genuine global conflict | Triple-slash directive |
| Package installed but types not found | Add to `types` array first |

## How This Looked in Our Package

Our final `tsconfig.json` for `api`:

```json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "noEmit": true,
    "types": ["@types/bun", "./worker-configuration.d.ts"]
  },
  "include": ["src"]
}
```

Two entries in `types`. That's the whole story. The worker gets its Cloudflare bindings. The CLI script gets its Bun globals. No triple-slash directives anywhere in the source.

---

## The Architecture in Plain English

```
node_modules/
  @types/bun/           ← installed, but ignored unless listed
  @types/node/          ← installed, but ignored unless listed

tsconfig.json
  "types": [
    "@types/bun",                  ← ✅ included
    "./worker-configuration.d.ts"  ← ✅ included
  ]
  // everything else             ← excluded
```

The `types` array is the bouncer. If it exists, nothing gets in without being on the list.

---

**The Golden Rule**: If you're adding a triple-slash directive to a `.ts` source file, stop and check your tsconfig `types` array first. The directive is almost always treating a symptom instead of the cause.
